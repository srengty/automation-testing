Foundations

What Is Testing Strategy?

Testing strategy determines how we approach testing based on what we know about the system under test, focusing on either external behavior or internal structure.

Strategy Coverage Quality
0:15 - 0:20 | Core definition

Blackbox Testing

Functional Testing Approach

  • Tests functionality without examining internal code structure.
  • Based on specifications, requirements, and user expectations.
  • Focuses on input-output behavior and user scenarios.
  • Independent of implementation details.
0:20 - 0:25 | Blackbox fundamentals

Blackbox Testing

Key Techniques

  • Equivalence Partitioning: Group inputs into valid/invalid classes.
  • Boundary Value Analysis: Test at edges of input domains.
  • Decision Tables: Map complex business rules.
  • State Transition Testing: Test system behavior changes.
0:25 - 0:32 | Blackbox techniques

Whitebox Testing

Structural Testing Approach

  • Tests based on internal code structure and logic.
  • Examines control flow, data flow, and execution paths.
  • Ensures all code paths are tested and validated.
  • Requires knowledge of implementation details.
0:32 - 0:38 | Whitebox fundamentals

Whitebox Testing

Coverage Criteria

  • Statement Coverage: Execute every line of code.
  • Branch Coverage: Test all decision outcomes (true/false).
  • Path Coverage: Execute all possible execution paths.
  • Condition Coverage: Test all boolean sub-expressions.

When to Use

Choosing the Right Approach

  • Blackbox: User acceptance, system testing, requirements validation.
  • Whitebox: Unit testing, code review, security testing.
  • Combined: Most effective testing uses both approaches.
  • Gray-box: Hybrid approach combining both strategies.

Part 2: Blackbox Testing

Equivalence Partitioning Technique

Divide input domain into classes where all values should behave similarly:

  • Valid partitions: Input that should work correctly
  • Invalid partitions: Input that should be rejected
  • Rule: Test at least one value from each partition
Systematic Efficient Complete

Blackbox Technique

Boundary Value Analysis

Test values at the edges of input domains where errors commonly occur:

  • Just below minimum: Test value just below valid range
  • Minimum value: Test lowest valid input
  • Just above minimum: Test value just above minimum
  • Maximum value: Test highest valid input
  • Just above maximum: Test value just above valid range

Part 3: Whitebox Testing

Understanding Code Coverage

Code coverage measures how much of your source code is tested:

  • Statement Coverage: % of code lines executed
  • Branch Coverage: % of decision points tested
  • Path Coverage: % of execution paths tested
  • Function Coverage: % of functions called
1:48 - 1:55 | Coverage fundamentals

Whitebox Example

Statement Coverage Analysis

Code Example:


public int divide(int a, int b) {
    if (b == 0) {               // Line 1
        throw new Exception();  // Line 2
    }
    int result = a / b;         // Line 3
    return result;              // Line 4
}

Test: divide(10, 2) covers lines 1, 3, 4 = 75% coverage

Missing: Line 2 (exception path)

1:55 - 2:02 | Statement coverage

Whitebox Example

Branch Coverage Analysis

Same Code with Branch Focus:


public int divide(int a, int b) {
    if (b == 0) {         // Decision point: TRUE or FALSE?
        throw new Exception();  
    }
    int result = a / b;         
    return result;              
}

Need 2 tests for 100% branch coverage:

  • Test 1: divide(10, 2) → covers FALSE branch
  • Test 2: divide(10, 0) → covers TRUE branch

Whitebox Advanced

Cyclomatic Complexity

Measure of code complexity based on decision points:

  • Formula: V(G) = E - N + 2P

Formula Elements:

  • V(G): Cyclomatic complexity of graph G
  • E: Number of edges in the control flow graph
  • N: Number of nodes in the control flow graph
  • P: Number of connected components (usually 1)
  • Simple: Count decision points + 1
  • if, while, for, case: Each adds +1 to complexity
  • Guideline: Complexity > 10 suggests refactoring
Maintainable Testable Quality

Best Practices

Effective Testing Strategy

  • Start with blackbox tests based on requirements.
  • Add whitebox tests to ensure code coverage.
  • Use static analysis tools for whitebox insights.
  • Document test rationale for both approaches.
  • Maintain balance between coverage and effort.

Summary

Key Takeaways

  • Blackbox testing validates functional requirements.
  • Whitebox testing ensures code structure quality.
  • Both approaches are complementary, not competitive.
  • Choose strategy based on testing goals and context.
  • Effective testing combines multiple approaches.
